# Project instructions

## 唯一研究问题

给定一幅秧苗图像，算法能否在没有人工关键点定义和关键点坐标标签的情况下，根据当前植株的结构与形态，自主确定用于表型计算的关键点位置、数量及其稳定性？

## 不可偏离的边界

- 本工程只服务于当前“自适应关键点发现”小论文。
- 不恢复任何预定义固定关键点体系，也不以旧关键点标签训练或评测。
- 不讨论跨作物迁移；该方向已封存。
- `G1′` 是当前结构候选生成基线的工作代号，不宣称它已经是最终模型。
- DINOv2 是独立视觉骨干/表征模型；在本研究中提供局部语义证据，不得描述成 Swin-T 的内部模块。
- 最终价值由表型计算是否更准、更稳来判断，而不是由“点越少越好”判断。
- 未完成独立测试、消融和多随机种子复跑前，不得把候选结果写成已验证结论。

## 当前技术主线

1. 纯净、多时期的单株整株图像；地上部与种子/基部/根使用独立掩膜记录，不再维护 shoot-only 图像版本。
2. G1′：前景/骨架端点与分叉点/曲率候选，加上 DINOv2 局部表征证据。
3. 将候选点连接、排序为茎叶路径，进行样条重建和表型计算。
4. 以 G1′跨增强一致的高置信结果作为伪标签，训练可学习的动态关键点模型。
5. 用稳定性、路径重建和表型误差进行比较与消融。

## 数据规则

- 原图与背景标准化图必须分开记录，任何处理图都不能冒充原始采集图。
- 数据划分按原始植株/采集组完成，同一植株的裁剪或增强不得跨 train/val/test。
- 自动背景标准化图必须经过可视化复核；带棋盘格、灰块、断裂或明显残缺的图不得进入正式测试集。
- 时期标签只用于覆盖度审计和分层划分，不作为关键点监督。

## 设备规则

- 当前本机无 CUDA，只做数据处理、静态检查和小样本 CPU 冒烟测试。
- 深度训练在 VS Code 连接的远程 GPU 服务器执行。
- 远程开始前先运行项目根目录的 `remote_gpu_check.py`。

## 固定工作方式（账号与 Skills 无关）

- 不依赖某个账号是否安装 `Superpowers`。即使该 Skill 不可用，也必须执行本节的同等流程。
- 涉及研究问题、方法架构、教师目标、数据划分、评价指标或论文主结论的改变时：先检查现有证据，提出 2–3 个方案及利弊，明确推荐项，得到用户确认后再实现。
- 已经批准的常规工作可以直接执行，包括：运行既有训练/评估脚本、按锁定配置复跑随机种子、执行既定消融、整理日志、做静态检查与失败样本审计。
- 修改代码前先检查 `git status`、最近提交、`PROJECT_STATE.md` 和相关输出；不得凭聊天印象重建方案。
- 每项代码修改都要有与风险相称的验证。训练入口至少完成 dry-run 或最小反向传播；结果脚本至少核对样本数、数据划分、输出字段和失败可见性。
- 不因某次指标好看就宣布成功。必须同时报告失败样本、适用边界、尚未完成的人工验证和不能推出的结论。
- 每次形成新的有效结果、失败结论或路线变更后，更新 `PROJECT_STATE.md`；需要跨账号保留的内容不能只留在聊天中。

## 数据选择与报告纪律

- `train` 与 `auxiliary` 可用于训练和自动教师目标；`val` 只用于模型/阈值选择；`test` 不得用于调参或决定是否修改算法。
- 已经查看过的 test 结果必须如实记录，不得再把 test 描述为完全未接触的选择集。
- 路径连通或程序无报错不等于语义正确；漏叶、错枝、基部错判必须进入失败样本审计。
- 在人工路径语义和人工表型参考未完成前，`phenotype_accuracy_status` 必须保持 `pending`，不得报告表型准确率已经通过。
- checkpoint、第三方源码和大体积原始数据不进入 Git；配置、日志摘要、关键 CSV、失败样本图和结论文档应进入 Git。

- 重新开放跨作物迁移方向；

- 更换核心视觉骨干、教师目标定义或主要评价标准；

- 对论文作超出当前小论文范围的结构扩张。
